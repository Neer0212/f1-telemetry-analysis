# F1 Telemetry Analysis

A Python toolkit for analyzing Formula 1 telemetry and timing data using the [FastF1](https://github.com/theOehrly/Fast-F1) API. Covers single-session deep dives, season-long championship analysis, and driver head-to-head telemetry comparisons — all packaged as a reusable library with runnable scripts that generate charts and Markdown reports.

![Race pace example](docs/images/race_pace_demo.png)

> The chart above is generated by this project's own plotting code (`plot_race_pace`), rendered with synthetic data to illustrate the output format. Run the scripts in `scripts/` against real session data to reproduce charts like this for any Grand Prix.

## What this does

- **Single-session analysis** — fastest laps, race pace consistency, tire strategy reconstruction, for any practice/qualifying/race session since 2018.
- **Driver head-to-head telemetry** — speed trace overlays, cumulative time-delta along the lap, and throttle/brake comparisons between any two drivers' laps.
- **Season-level analysis** — round-by-round championship progression for both drivers' and constructors' standings, pulled from the Ergast API.
- **Track speed maps** — the circuit outline colored by speed at every point on track.
- **Markdown session reports** — a clean, shareable summary (fastest laps, pace stats, tire strategy, weather) generated for any session.

## Project structure

```
f1-telemetry-analysis/
├── f1_analysis/                 # the installable package
│   ├── core/
│   │   ├── session_loader.py    # FastF1 session loading + caching
│   │   ├── lap_analysis.py      # lap cleaning, pace stats, stint summaries
│   │   ├── telemetry.py         # per-lap telemetry, driver-vs-driver comparison
│   │   └── season.py            # season schedule + championship standings (Ergast)
│   ├── visualization/
│   │   ├── style.py             # FastF1 team/driver/compound color helpers
│   │   └── plots.py             # all chart-generating functions
│   └── reports/
│       └── session_report.py    # Markdown report generation
├── scripts/                     # runnable CLI entry points
│   ├── 01_session_deep_dive.py
│   ├── 02_driver_head_to_head.py
│   ├── 03_season_championship.py
│   └── 04_track_speed_map.py
├── tests/                       # offline unit tests (no network required)
├── outputs/                     # generated charts/reports land here (gitignored)
├── cache/                       # FastF1's local data cache (gitignored)
└── docs/images/                 # demo images used in this README
```

The split exists so each piece is reusable on its own: `f1_analysis.core` has no plotting dependencies if you just want the data, `f1_analysis.visualization` builds on `core` for charts, and `scripts/` wires both together into things you can run from the command line.

## Installation

Requires Python 3.9+.

```bash
git clone https://github.com/<your-username>/f1-telemetry-analysis.git
cd f1-telemetry-analysis
pip install -r requirements.txt
pip install -e .
```

The `-e .` install makes the `f1_analysis` package importable from anywhere (including your own notebooks/scripts) while you keep editing the source.

## Quick start

```python
from f1_analysis.core.session_loader import load_session
from f1_analysis.core.lap_analysis import clean_lap_times, fastest_laps_by_driver
from f1_analysis.visualization.style import apply_f1_style
from f1_analysis.visualization.plots import plot_race_pace

apply_f1_style()

session = load_session(2024, "Monza", "R")          # year, Grand Prix, session
clean = clean_lap_times(session.laps)                # drop pit/inaccurate laps
print(fastest_laps_by_driver(clean).head())

fig = plot_race_pace(session, ["VER", "LEC", "NOR"])
fig.savefig("race_pace.png")
```

The first time you load a given session, FastF1 downloads and caches it under `cache/` (configurable). Every subsequent load of that same session is fast and offline.

## Using the scripts

Each script is a standalone CLI tool. Outputs land in `outputs/charts/` and `outputs/reports/`.

**1. Single-session deep dive** — fastest laps, race pace, tire strategy, plus a Markdown report:
```bash
python scripts/01_session_deep_dive.py --year 2024 --gp Monza --session R
python scripts/01_session_deep_dive.py --year 2023 --gp "Abu Dhabi" --session Q --drivers VER LEC HAM
```

**2. Driver head-to-head** — speed trace, time delta, and throttle/brake comparison for two drivers' laps:
```bash
python scripts/02_driver_head_to_head.py --year 2024 --gp Monza --session Q --driver-a VER --driver-b LEC
```

**3. Season championship progression** — round-by-round standings for an entire season:
```bash
python scripts/03_season_championship.py --year 2024
```
This makes one network request per round to the Ergast API (bundled in FastF1), so a full season takes a couple of minutes. Results are cached to CSV in `outputs/reports/` so you don't need to re-fetch them.

**4. Track speed map** — circuit outline colored by speed:
```bash
python scripts/04_track_speed_map.py --year 2024 --gp Monaco --session Q --driver LEC
```

Run any script with `--help` to see all options.

## Data source and caching

This project is built entirely on [FastF1](https://docs.fastf1.dev/), which sources data from F1's live timing feed (lap times, car telemetry, weather) and the Ergast API (historical results, standings). Data is available from the 2018 season onward; telemetry detail and availability can vary for older or non-Grand-Prix sessions.

Caching is handled by `f1_analysis.core.session_loader.enable_cache()`, called automatically the first time you load a session. Cached data lives in `cache/` and is gitignored — expect tens of MB per session with full telemetry, so don't commit it.

## Running tests

```bash
pip install -r requirements-dev.txt
pytest
```

Tests run fully offline: they exercise the pandas-level logic (lap cleaning, pace stats, stint summaries) and the plotting functions against synthetic, hand-built data rather than live FastF1 sessions, so they don't depend on network access or data availability.

## Notes and limitations

- The `delta_time` calculation used in head-to-head comparisons (via FastF1's `utils.delta_time`) is a useful approximation but isn't perfectly precise — FastF1's own docs note it can diverge slightly from sector-time-derived deltas. Treat it as directionally accurate rather than to-the-millisecond exact.
- Two teammates share the same team color; the plotting functions use `get_driver_style` (color + line style) rather than plain color to keep them visually distinguishable.
- Season-level standings functions make one API call per round, so building a full-season progression involves ~20+ requests. This is intentional (Ergast doesn't expose a single "progression" endpoint) but worth knowing before running it repeatedly.
